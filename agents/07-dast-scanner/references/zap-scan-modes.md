# OWASP ZAP Scan Modes Reference

## Overview

OWASP ZAP (Zed Attack Proxy) is run via Docker in the GRIMSEC DAST pipeline using ProjectDiscovery's officially maintained image: `ghcr.io/zaproxy/zaproxy:stable`.

ZAP exposes three scan scripts corresponding to three distinct testing strategies. Choosing the correct mode is critical for balancing coverage, scan time, and risk to the target application.

---

## Mode 1: Baseline Scan (`zap-baseline.py`)

**Purpose:** Passive-only scan — no active attack attempts. Spider + passive analysis only.

**Duration:** ~1–5 minutes  
**Safe for production:** Yes  
**Active attacks:** No  
**CI/CD recommended:** Yes  

### What it does
1. Spiders the target URL to discover pages and endpoints
2. Applies passive scan rules to all observed traffic (no new requests generated by the scanner itself)
3. Reports on issues observable from response headers and bodies

### What it finds
- Missing security headers (Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, HSTS)
- Cookie flags not set (Secure, HttpOnly, SameSite)
- Information disclosure in HTTP headers (Server version, X-Powered-By)
- Mixed content (HTTP resources on HTTPS pages)
- Private IP addresses exposed in responses
- Timestamp disclosure
- HTTPS to HTTP redirect chains

### What it does NOT find
- XSS, SQLi, or any injection-class vulnerabilities (requires active scan)
- Authentication bypass
- Business logic flaws

### Docker command
```bash
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  ghcr.io/zaproxy/zaproxy:stable \
  python /zap/zap-baseline.py \
  -t https://example.com \
  -r zap-report.html \
  -J zap-report.json \
  -x zap-report.xml
```

### When to use
- Every CI/CD build as a quality gate
- Quick pre-deployment check
- Scanning production systems where active probing is prohibited
- When you have < 5 minutes for security testing

## Mode 2: Full Scan (`zap-full-scan.py`)

**Purpose:** Active + passive scan. Comprehensive dynamic testing with active attack payloads.

**Duration:** 20–90 minutes (depends on application size)  
**Safe for production:** No — generates actual attack payloads  
**Active attacks:** Yes  
**CI/CD recommended:** Only in staging/pre-prod pipelines  

### What it does
1. Spiders the target to build a site map
2. AJAX spider for JavaScript-rendered content
3. Passive scan on all discovered content
4. Active scan with attack payloads against all discovered endpoints

### What it finds (active scan rules)
| Vulnerability Class | ZAP Rule | CWE |
|--------------------|----------|-----|
| SQL Injection | 40018, 40019, 40020, 40021, 40022 | CWE-89 |
| Reflected XSS | 40012, 40014 | CWE-79 |
| Path Traversal | 6 | CWE-22 |
| Remote File Inclusion | 7 | CWE-98 |
| Server-Side Include | 40009 | CWE-97 |
| LDAP Injection | 40015 | CWE-90 |
| XPath Injection | 40010 | CWE-643 |
| XML External Entity | 90023 | CWE-611 |
| CRLF Injection | 40003 | CWE-93 |
| Parameter Tampering | 40008 | CWE-472 |
| Buffer Overflow | 30001 | CWE-120 |
| Integer Overflow | 30003 | CWE-190 |
| Remote OS Command Injection | 20018 | CWE-78 |

Plus all passive scan rules from baseline mode.

### Docker command
```bash
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  ghcr.io/zaproxy/zaproxy:stable \
  python /zap/zap-full-scan.py \
  -t https://example.com \
  -r zap-report.html \
  -J zap-report.json \
  -x zap-report.xml \
  -I
```

### Authentication (form-based)
For authenticated scans, use a ZAP context file:
```bash
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  -v $(pwd)/auth-context:/zap/context \
  ghcr.io/zaproxy/zaproxy:stable \
  python /zap/zap-full-scan.py \
  -t https://example.com \
  -n /zap/context/auth.context \
  -U "testuser" \
  -r zap-report.html
```

### When to use
- Pre-release security gates in staging environments
- Penetration testing of dedicated test instances
- Quarterly security assessments
- When you need OWASP Top 10 coverage with active verification

## Mode 3: API Scan (`zap-api-scan.py`)

**Purpose:** OpenAPI/Swagger/SOAP/GraphQL-aware active scan targeting API endpoints directly.

**Duration:** 10–40 minutes (depends on API surface)  
**Safe for production:** No — sends attack payloads  
**Active attacks:** Yes (on all API endpoints from spec)  
**CI/CD recommended:** Staging only  

### What it does
1. Imports API definition (OpenAPI 2/3, SOAP WSDL, or GraphQL introspection)
2. Generates requests for all defined endpoints with appropriate parameters
3. Applies active and passive scan rules to each endpoint
4. Does NOT rely on spidering — uses the spec as the ground truth

### Supported API spec formats
| Format | Flag | Example |
|--------|------|---------|
| OpenAPI 3.x (JSON or YAML) | `-f openapi` | `openapi.yaml` |
| Swagger 2.x (JSON or YAML) | `-f openapi` | `swagger.json` |
| SOAP/WSDL | `-f soap` | `service.wsdl` |
| GraphQL | `-f graphql` | `http://api/graphql` |

### Docker command (OpenAPI spec)
```bash
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  ghcr.io/zaproxy/zaproxy:stable \
  python /zap/zap-api-scan.py \
  -t https://api.example.com \
  -f openapi \
  -r zap-report.html \
  -J zap-report.json \
  -x zap-report.xml \
  -I
```

### API spec from URL (no file needed)
```bash
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  ghcr.io/zaproxy/zaproxy:stable \
  python /zap/zap-api-scan.py \
  -t https://api.example.com/v1/openapi.json \
  -f openapi \
  -r zap-report.html
```

### When to use
- REST APIs with OpenAPI/Swagger spec
- Microservices with defined API contracts
- GraphQL endpoints
- SOAP web services
- When the application has no traditional web UI to spider

## ZAP Exit Codes

| Code | Meaning | GRIMSEC Action |
|------|---------|---------------|
| 0 | No alerts found | Pass |
| 1 | Informational alerts only | Pass |
| 2 | Warning-level alerts | Flag for review |
| 3 | Failure (scan error or high+ alerts) | Investigate |

Note: `run-zap.py` uses `-I` flag (ignore rules) and parses report files directly, so exit codes are not used as the primary pass/fail signal.

## ZAP Alert Risk Levels

| ZAP Risk Code | ZAP Label | GRIMSEC Severity |
|---------------|-----------|-----------------|
| 3 | High | high |
| 2 | Medium | medium |
| 1 | Low | low |
| 0 | Informational | info |

The `run-zap.py` wrapper upgrades severity for specific alert names:
- "SQL Injection" → critical
- "Remote OS Command Injection" → critical
- "Stored XSS" / "Persistent XSS" → high
- "SSRF" → high

## ZAP Scan Tuning

### Adjusting active scan strength
ZAP's active scan has configurable strength and threshold. Use a context file or `-config` flags:

```bash
# High strength (more thorough, slower, more false positives)
-config scanner.attackStrength=HIGH

# Medium threshold (default — balances FP rate)
-config scanner.alertThreshold=MEDIUM
```

### Excluding paths from scan
To exclude logout endpoints or other destructive actions:
```bash
-config spider.excludeRegex=".*logout.*|.*delete.*|.*reset-password.*"
```

### Increasing spider depth
```bash
-config spider.maxDepth=10
-config spider.maxChildren=50
```

### Timeout configuration
ZAP Docker entrypoints support `-T <minutes>` for total scan timeout:
```bash
python /zap/zap-full-scan.py -t https://example.com -T 60
```

## Integration with GRIMSEC Pipeline

The `run-zap.py` script handles:
1. Docker invocation with volume mounting for report extraction
2. XML/JSON report parsing into normalized finding dicts
3. Severity normalization and CWE/OWASP mapping
4. Structured JSON output compatible with `vulnerability-context-enricher` input format

For API-only targets where no OpenAPI spec exists, use the full scan with `--mode full` and set a conservative spider depth to avoid explosive crawl times.
