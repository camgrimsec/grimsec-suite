# Security PR Template

Use this template when creating pull requests for security remediation.

## PR Title Format

```
fix(security): [CVE-XXXX-XXXXX] Remediate {vulnerability type} in {component}
```

Examples:
- `fix(security): [CVE-2024-38816] Remediate path traversal in spring-webmvc`
- `fix(security): [CVE-2024-4068] Upgrade braces to fix ReDoS vulnerability`
- `fix(security): Remove exposed API key in config/settings.py`

## PR Description Template

```markdown
## 🔒 Security Fix: {Brief Description}

### Vulnerability Summary

| Field | Value |
|-------|-------|
| **CVE** | {CVE-XXXX-XXXXX or "N/A"} |
| **CWE** | {CWE-XXX: Category Name} |
| **CVSS Score** | {X.X} ({severity}) |
| **Real Risk Score** | {X}/10 ({classification}) |
| **Affected Component** | {package@version or file:line} |

### What Is This Vulnerability?

{2-3 sentences explaining the vulnerability in plain language. What could an attacker do?
Avoid overly technical jargon — write for a senior developer, not a security researcher.}

### Why It Matters for This Project

{2-3 sentences specific to THIS application's context. Reference the application type,
how data flows through the affected component, and why this vulnerability is relevant
given the project's architecture and user base.}

### What This PR Changes

{Bullet list of specific changes:}
- Updated `{file}`: {description of change}
- Updated `{dependency file}`: Upgraded {package} from {old_version} to {new_version}

### Breaking Change Risk

{Assessment of whether this fix could break existing functionality:}
- **Risk Level:** {None | Low | Medium | High}
- **Details:** {Explain any potential breaking changes and how to verify}

### Verification Steps

To verify this fix:

1. {Step 1 — e.g., "Run the existing test suite: `npm test`"}
2. {Step 2 — e.g., "Verify the affected endpoint still works: `curl -X POST /api/endpoint`"}
3. {Step 3 — e.g., "Confirm the vulnerability is resolved: `trivy fs . --severity CRITICAL`"}

### References

- [{CVE-XXXX-XXXXX} — NVD](https://nvd.nist.gov/vuln/detail/{CVE-ID})
- [{CWE-XXX}](https://cwe.mitre.org/data/definitions/{CWE-number}.html)
- [{Advisory link if applicable}]({url})
- [Fix commit in upstream]({url if applicable})

---

*This PR was generated by an automated DevSecOps analysis pipeline. The vulnerability
was identified through scanning and verified through contextual reachability analysis.
Please review the changes carefully before merging.*

*If you have questions about the analysis methodology or need additional context,
please comment on this PR.*
```

## Tone Guidelines

- **Collaborative, not adversarial.** You're helping, not shaming.
- **Professional.** This represents you as a DevSecOps consultant.
- **Specific.** Generic "update dependencies" PRs are spam. Every PR must explain WHY.
- **Respectful of maintainers' time.** Keep it concise. Link to details rather than dumping them.
- **Honest about limitations.** If automated analysis has uncertainty, say so.

## Commit Message Format

Follow conventional commits:

```
fix(security): {brief description}

{CVE-ID}: {one-line explanation}

Affected: {component}@{version}
Fixed: {component}@{new_version}
Real Risk Score: {X}/10

Signed-off-by: {Your Name} <{your-email}>
```

## Branch Naming

```
security/fix-{cve-id}
```

Or for non-CVE findings:

```
security/fix-{finding-type}-{component}
```

Examples:
- `security/fix-CVE-2024-38816`
- `security/fix-secrets-config-settings`
- `security/fix-sqli-user-controller`
